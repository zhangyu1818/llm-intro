# i18n 设计方案 · 看懂大模型

- 日期：2026-05-17
- 状态：Draft，待实施
- 目标：为现有的纯前端 SSG 站点加上 zh / en 双语支持，组件结构唯一，文案分语言存放，部署形态保持 GitHub Pages 单仓单 build。

## 背景

项目当前形态：

- React 19 + Vite + TypeScript + Tailwind v4，纯前端静态站
- 17 个章节组件（`src/sections/ChapterXX.tsx`）+ `Hero` + `Outro` + `Navigator`
- 文案高度嵌入 JSX，含 `<Mark>`、`<Pull>`、`<strong>`、句中插入等富文本
- 全局元数据 `src/data/chapters.ts` 持有章节标题/副标题
- 部署到 GitHub Pages（`zhangyu1818.github.io/llm-intro/`）

引入英文版的核心约束：

1. **组件唯一**：不接受为每种语言复制一份 Chapter 组件。
2. **富文本保真**：`<Mark>` / `<Pull>` 等组件嵌入必须保留，不接受 `<Trans i18nKey>` 占位语法。
3. **URL 真路径**：英文走 `/en/`，可分享、可被搜索引擎收录。
4. **零额外运行时**：项目体量不需要 i18next 这类完整方案。

## 架构决策

### 决策一：每章 colocated i18n 文件

每个 Chapter 旁边放 `ChapterXX.i18n.tsx`，默认导出 `{ zh, en }`：

```tsx
// sections/Chapter00.i18n.tsx
const zh = {
  title: <><span className="block">它不在脑子里</span><span className="block">记着你</span></>,
  caption: '它更像手里拿着你递过去那叠材料，当场把答案拼出来的人',
  intro: <>第一次接触大模型，先别急着学怎么提问。<Mark>它并不是一个在「心里」一直惦记你、记着你、懂你的人。</Mark></>,
  materials: [/* 互动卡片数据 */],
  // ...
}

const en: typeof zh = { /* 同结构，英文文案 */ }

export default { zh, en }
```

组件文件保持唯一一份，仅替换文案来源：

```tsx
import content from './Chapter00.i18n'
import { useLocale } from '../i18n/LocaleContext'

export function Chapter00() {
  const t = content[useLocale()]
  return <Section title={t.title} caption={t.caption}>...</Section>
}
```

**为何这样选**：

- 文案与组件相邻，作者改一章时左右两个文件即可，PR diff 局部
- `typeof zh` 让 en 必须结构对齐，**漏一个字段就 TS 编译失败**
- ReactNode 作为字典 value，富文本无损保留
- 不需要任何运行时 i18n 库

**拒绝的替代方案**：

- 全局单字典（i18next 风格）：拆 key 后路径深、长文 ReactNode 集中在一个 4000 行文件，维护痛
- MDX 多语言：Chapter 内大量交互组件（material 卡片、state、按钮），MDX 不能管，等于要双轨制
- i18next + `<Trans>`：长文 + 多富文本场景下，`<1>`/`<2>` 占位语法比直接写 JSX 更难读

### 决策二：Vite 多入口生成两份 HTML

构建产物：

```
dist/
  index.html          # zh
  en/
    index.html        # en
  assets/             # 共享 chunk
```

`vite.config.ts`：

```ts
build: {
  rollupOptions: {
    input: {
      zh: resolve(__dirname, 'index.html'),
      en: resolve(__dirname, 'en/index.html'),
    },
  },
}
```

两份 HTML 镜像同一份结构，只在以下处不同：

| | `index.html` | `en/index.html` |
|---|---|---|
| `<html lang>` | `zh-CN` | `en` |
| `<title>` | 看懂大模型 · A Vivid Guide | A Vivid Guide to LLMs |
| `<meta name="description">` | 中文描述 | 英文描述 |
| `<script src>` | `/src/main.zh.tsx` | `/src/main.en.tsx` |
| `<link rel="alternate" hreflang>` | 互指（双向） | 互指（双向） |

**为何这样选**：

- 爬虫看到的就是真静态对应语言的 HTML，SEO 与可分享性都对
- GH Pages 部署 `dist/` 即可，无需 404 fallback 等技巧
- 共享 JS chunk，bundle 不会爆增；用户语言的字典靠入口分流，理论上可 tree-shake

**拒绝的替代方案**：

- 单 SPA + `location.pathname` 分流：GH Pages 在 `/en/` 路径需 404 → index.html 跳板，且初始 HTML 只有一种语言，SEO 损失
- SSR 预渲染：引入服务端构建复杂度，对静态阅读站收益太低

### 决策三：极薄的 LocaleContext

```tsx
// src/i18n/LocaleContext.tsx
import { createContext, useContext } from 'react'

export type Locale = 'zh' | 'en'
const Ctx = createContext<Locale>('zh')
export const LocaleProvider = Ctx.Provider
export const useLocale = () => useContext(Ctx)
```

入口注入：

```tsx
// src/main.zh.tsx
createRoot(root).render(
  <LocaleProvider value="zh"><App /></LocaleProvider>
)

// src/main.en.tsx 同理，value="en"
```

**为何这样选**：

- 不需要 reducer/dispatch，locale 由入口编译期决定，运行时不变
- 没有切换 state，因为切换 = 跳到另一入口（真跳转）

## 全局零散文案

不属于任何具体章节的文字，集中到 `src/i18n/global.{zh,en}.ts`：

- Hero：`A Vivid Guide · 2026`、英雄标题/副文、`17 章 · 从零建立直觉` 等 chips、`开始 · scroll`、底部 hint
- Navigator：`回到开头` aria-label、品牌名（"看懂大模型"）
- Outro：marquee `lines: string[]`、`The End · But just the beginning`、底部标题、底部副文、`A vivid guide to LLMs · 2026`、`源码 · GitHub`、`回到开头 ↑`
- chapter 元数据：`title` 与 `caption` 改为按 locale 取
- LangSwitcher：按钮文字（如 `ZH` / `EN` / `中` / `EN`），最终样式实施时再敲定

`chapters.ts` 的 `id` / `index` / `accent` 保持静态共享；`title` / `caption` 通过函数从 global 字典取：

```ts
// src/data/chapters.ts
export const chapterMeta = [
  { id: 'hero', index: '00', accent: '--color-sun' },
  { id: 'ch-0', index: '01', accent: '--color-ember' },
  // ...
]

// 在使用处：
const meta = chapterMeta.map((m) => ({ ...m, ...globalDict.chapters[m.id] }))
```

## LangSwitcher

挂载位置：`Navigator` header 右侧，与章节进度指示并列。

形态：两段链接 `ZH · EN`，当前 active 加重。

行为：

```tsx
const otherLang = useLocale() === 'zh' ? 'en' : 'zh'
const href = otherLang === 'en'
  ? `/en/${window.location.hash}`
  : `/${window.location.hash}`
```

跨入口必须真跳转（不是 `history.pushState`），但 `#ch-3` 会保留，让用户回到对应章节。GH Pages 子路径部署时需要兼容 `base`（见下）。

## GitHub Pages base 路径与 404 fallback

项目部署到 `zhangyu1818.github.io/llm-intro/`。已有配置：

- `vite.config.ts`：`base: mode === 'production' ? '/llm-intro/' : '/'`
- workflow：`cp dist/index.html dist/404.html`，作为 GH Pages 的 SPA fallback

i18n 后两处的处理：

- **base 路径**：vite 多入口在 `base = '/llm-intro/'` 下产出
  - `/llm-intro/` → `dist/index.html`（zh）
  - `/llm-intro/en/` → `dist/en/index.html`（en）
  - 共享 `dist/assets/*`，Vite 自动处理两个 HTML 的相对路径
- **LangSwitcher href** 用 `import.meta.env.BASE_URL` 拼接：
  ```ts
  const base = import.meta.env.BASE_URL  // '/llm-intro/' 或 '/'
  const href = otherLang === 'en'
    ? `${base}en/${window.location.hash}`
    : `${base}${window.location.hash}`
  ```
- **404 fallback 维持现状**（复制中文 `index.html` 为 `404.html`）：
  - GH Pages 只有一个全局 404，无法按路径前缀区分语言
  - 访问 `/llm-intro/en/wrong-path` 会回退到中文 404 页，用户点 EN 切换即可恢复
  - 第一版接受这个轻微体验损耗；若以后需要双语 404，可写一个根据 `location.pathname` 在客户端跳转的 404.html 模板

## 失败兜底

- **缺翻译**：`en: typeof zh` 在编译期把所有缺失/类型不匹配抓住，运行时不会出现 `undefined`。
- **错链接**：访问 `/en/#ch-99`（不存在的 hash）→ React mount 后没有匹配 section，自然滚到顶部，无错。
- **入口资源缺失**：构建期 vite 会校验入口 HTML 存在与否，缺失直接构建失败。

## SEO

每份 HTML head 内：

```html
<html lang="zh-CN">  <!-- 或 "en" -->
<head>
  <title>{对应语言}</title>
  <meta name="description" content="{对应语言}" />
  <link rel="canonical" href="https://zhangyu1818.github.io/llm-intro/" />
  <link rel="alternate" hreflang="zh-CN" href="https://zhangyu1818.github.io/llm-intro/" />
  <link rel="alternate" hreflang="en" href="https://zhangyu1818.github.io/llm-intro/en/" />
  <link rel="alternate" hreflang="x-default" href="https://zhangyu1818.github.io/llm-intro/" />
</head>
```

`canonical` 在两份 HTML 中分别指向自身 URL。

## 不做的事（YAGNI）

- 不引入 i18next / react-intl / formatjs
- 不做浏览器语言自动跳转（默认 `/` 是中文，切换由用户主动；第一版后再评估）
- 不做语言包动态懒加载（两个入口已天然拆开）
- 不做服务端/构建时预渲染（用户内容已经是静态 ReactNode）
- 不做翻译工作流工具（个人项目，作者手翻）
- 不做 RTL / 复数 / 日期数字 locale 格式化

## 实施分阶段

后续 `writing-plans` 会展开为具体可执行 plan。这里仅列里程碑：

1. **地基**：新建 `LocaleContext.tsx` + 双 `main.{zh,en}.tsx` + 双 HTML + `vite.config.ts` 多入口，跑通构建

3. **LangSwitcher**：挂上 Navigator，链接拼接 base + hash
4. **逐章迁移**：Chapter00 → Chapter16，每章一次 commit；每章迁移即完整翻译一次英文
5. **HTML SEO 收尾**：title / description / hreflang / canonical
6. **可选优化**：英文页字体瘦身（去掉 Noto Sans SC / Noto Serif SC / Fraunces 的中文子集）

## 验收标准

- `pnpm build` 产出 `dist/index.html` 与 `dist/en/index.html` 两份 HTML
- 中文页 `<html lang="zh-CN">`，英文页 `<html lang="en">`
- `/` 切换到 `/en/` 保留当前 `#chapter-id`
- 任意章节漏翻一字段 → TS 编译失败
- Lighthouse SEO 两份页面 ≥ 95
- 视觉与现有中文版无回归（除新增的 LangSwitcher）
